# Тестовое задание: API организационной структуры

## Модели

### Department — подразделение

| Поле        | Тип           | Описание                                      |
|-------------|---------------|-----------------------------------------------|
| `id`        | int           | Первичный ключ                                |
| `name`      | str           | Название (не пустое)                          |
| `parent_id` | int \| null   | FK на Department, позволяет строить дерево    |
| `created_at`| datetime      | Дата создания                                 |

### Employee — сотрудник

| Поле            | Тип         | Описание                        |
|-----------------|-------------|---------------------------------|
| `id`            | int         | Первичный ключ                  |
| `department_id` | int         | FK на Department                |
| `full_name`     | str         | Полное имя (не пустое)          |
| `position`      | str         | Должность (не пустая)           |
| `hired_at`      | date \| null| Дата найма (опционально)        |
| `created_at`    | datetime    | Дата создания                   |

### Связи

- `Department` 1—N `Employee`
- `Department` 1—N `Department` (самоссылка через `parent_id`)

## Методы API

### 1. Создать подразделение

```
POST /api/v1/departments
```

**Body:**
```json
{
  "name": "string",
  "parent_id": 1
}
```

**Response:** `201 Created` — созданное подразделение

---

### 2. Создать сотрудника в подразделении

```
POST /api/v1/departments/{id}/employees
```

**Body:**
```json
{
  "full_name": "string",
  "position": "string",
  "hired_at": "2024-01-15"
}
```

**Response:** `201 Created` — созданный сотрудник

---

### 3. Получить подразделение (детали + сотрудники + поддерево)

```
GET /api/v1/departments/{id}
```

**Query параметры:**

| Параметр           | Тип  | По умолчанию | Описание                                    |
|--------------------|------|--------------|---------------------------------------------|
| `depth`            | int  | `1`          | Глубина вложенных подразделений (макс. 5)   |
| `include_employees`| bool | `true`       | Включать ли сотрудников в ответ             |

**Response:** `200 OK`
```json
{
  "id": 1,
  "name": "Engineering",
  "parent_id": null,
  "created_at": "2024-01-15T10:00:00Z",
  "employees": [...],
  "children": [...]
}
```

---

### 4. Переместить / переименовать подразделение

```
PATCH /api/v1/departments/{id}
```

**Body:**
```json
{
  "name": "string",
  "parent_id": 2,
  "clear_parent": false
}
```

**Response:** `200 OK` — обновлённое подразделение

---

### 5. Удалить подразделение

```
DELETE /api/v1/departments/{id}
```

**Query параметры:**

| Параметр                   | Тип    | Описание                                                  |
|----------------------------|--------|-----------------------------------------------------------|
| `mode`                     | string | Режим удаления: `cascade` или `reassign`                  |
| `reassign_to_department_id`| int    | ID отдела для перевода сотрудников (обязателен для `reassign`) |

**Режимы удаления:**
- `cascade` — удалить подразделение, всех сотрудников и все дочерние подразделения
- `reassign` — перевести сотрудников в `reassign_to_department_id`, затем удалить подразделение (атомарно)

**Response:** `204 No Content`

## Бизнес-логика и ограничения

### Подразделения
- Нельзя создать подразделение с несуществующим `parent_id` → `404 Not Found`
- `name` не пустой, длина 1..200, пробелы по краям триммируются
- В пределах одного `parent_id` названия должны быть уникальны → `409 Conflict`
- Нельзя сделать подразделение родителем самого себя → `409 Conflict`
- Нельзя создать цикл в дереве (переместить департамент внутрь своего поддерева) → `409 Conflict`
- При `mode=reassign` нельзя удалить отдел у которого есть дочерние подразделения → `409 Conflict`

### Сотрудники
- Нельзя создать сотрудника в несуществующем подразделении → `404 Not Found`
- `full_name` не пустой, длина 1..200
- `position` не пустой, длина 1..200

## Коды ответов

| Код  | Описание                                 |
|------|------------------------------------------|
| 200  | Успешный запрос                          |
| 201  | Сущность создана                         |
| 204  | Успешное удаление (без тела ответа)      |
| 400  | Неверный формат запроса                  |
| 404  | Сущность не найдена                      |
| 409  | Конфликт бизнес-правил                   |
| 422  | Невалидные входные данные                |
| 500  | Внутренняя ошибка сервера                |

## Технические требования

- **Язык:** Go
- **HTTP:** стандартный `net/http` (без сторонних роутеров)
- **ORM:** GORM
- **БД:** PostgreSQL
- **Миграции:** goose
- **Документация:** Swagger (swaggo/swag)
- **Контейнеризация:** Docker + docker-compose
